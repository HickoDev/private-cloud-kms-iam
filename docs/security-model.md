# Security Model

## Authentication

Users authenticate with email and password. Passwords are stored as bcrypt hashes. Successful login returns a JWT access token with an expiration time.

## Authorization

The backend enforces role-based access control. The frontend hides unavailable actions, but the backend remains the authority.

Roles:

- `ADMIN`: manages users and roles, can access all modules.
- `KEY_MANAGER`: creates, disables, and rotates keys.
- `KEY_USER`: encrypts and decrypts data.
- `AUDITOR`: reads audit logs and key metadata.

## Key Protection

KMS data keys are generated by the backend and encrypted before storage using `MASTER_KEY` from the environment.

Rules:

- raw key material is never returned by the API
- plaintext user data is not stored
- AES-256-GCM is used for encryption
- every encryption operation uses a fresh nonce
- old key versions remain available for decryption after rotation

## Auditability

Sensitive actions are written to audit logs:

- login success and failure
- user creation
- role assignment
- key creation
- key disable
- key rotation
- encryption and decryption
- denied access

## Academic Limitation

This project uses an environment variable as a simulated master key. A production system should use an HSM, secure vault, or cloud-native key protection service.
